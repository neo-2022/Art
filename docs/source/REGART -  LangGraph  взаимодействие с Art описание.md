описательная инструкция (без кода): как REGART (твой LangGraph-проект my_langgraph_agent) будет работать вместе с Art, какие механизмы взаимодействия закладываем, что уже реально есть в REGART, и что ещё нужно доделать, чтобы связка стала “100%” (то есть ничего не теряем, а если теряем — это тоже фиксируется как “дыра наблюдаемости”).

1) Главное: кто кому “принадлежит” и кто от кого не зависит
1.1 Art — отдельная система мониторинга

Art — это самостоятельный проект, который:

запускается раньше REGART и может работать без REGART;

собирает сигналы с любых проектов, не только LangGraph;

имеет свою “панель 0” (bootstrap UI), которая должна показывать “что сломано” даже если REGART не поднялся.

1.2 REGART — наблюдаемый проект (клиент/источник сигналов)

REGART — это набор сервисов/частей:

agent/ — LangGraph (Python),

ui/ — React UI (Vite),

agent/src/react_agent/ui_proxy.py — UI Proxy (FastAPI), который проксирует /api/* в LangGraph и даёт /ui/* для удобных действий.

Важно: UI Proxy — часть REGART, не Art. Art его не “использует”, а наблюдает и/или получает от него события.

2) Из чего состоит REGART (по текущему репозиторию) и где точки интеграции
2.1 UI Proxy (FastAPI)

UI Proxy делает две большие функции:

/api/* → проксирует запросы в LangGraph API (base URL задаётся LANGGRAPH_BASE_URL, по умолчанию http://127.0.0.1:2024).

/ui/* → “локальные удобные” эндпоинты: модели Ollama, запись MODEL в .env, статусы, старт/стоп (в текущем публичном коде — через tmux), probe tool_calls и т.д.

Если LangGraph недоступен, в текущем публичном коде UI Proxy возвращает простой JSON с "error": "LangGraph API недоступен" и base_url, статус 502.
(Ты ранее описывал более “богатый” формат upstream-ошибок с hint/actions — значит это либо локальные изменения, либо целевое состояние, которое нужно довнести в репозиторий/в Art-интеграцию.)

2.2 React UI + встроенный Debugger Level0/Level1 (внутри REGART)

В REGART уже есть “сквозной Debugger” как отдельный модуль ui/src/debugger/.

Level 0:

стартует до React render (ui/src/main.jsx явно делает initDebuggerLevel0() до createRoot(...).render(...)).

создаёт window.__DBG0__ как “источник правды” и хранит ring-буферы errors/events/network/snapshots, делает dedupe/throttle.

ставит ловушки window.error и unhandledrejection и при fatal/error показывает fallback overlay, независимый от React.

Level 1 (React):

ui/src/debugger/core.js — это адаптер: он сохраняет API “ошибочного ядра”, но реально пишет в Level 0, если тот доступен.

ui/src/App.jsx создаёт uiErrCore и при ошибках runStream пушит их в Debugger (scope=run, severity=warn|error), а панель отладки авто-открывается при error/fatal.

Debugger-спека прямо перечисляет каналы, которые должны проходить через Debugger: UI runtime, Network, Journal/spans, Models, Tools, Graph snapshots, и т.д., плюс правило “без хардкода” и “Jump только по данным”.

3) Как Art и REGART должны работать вместе (модель взаимодействия)

Ниже — целевой механизм. Он не делает REGART зависимым от Art, но делает Art “всевидящим” по отношению к REGART.

3.1 Порядок запуска (самая важная часть для “100%”)

Art Core запускается первым и поднимает:

ingest API (куда принимаем события),

snapshot/stream API,

“панель 0” (bootstrap UI Art).

Art Agent запускается на машине, где крутится REGART, и начинает собирать:

systemd/journald,

статусы юнитов,

проверки портов/процессов,

(при включении) OTLP.

Потом запускается REGART (LangGraph, UI Proxy, React UI).

Почему так: если React/Vite/Proxy вообще не стартуют, Art всё равно видит причины (журналы, падения, порт занят, crashloop) и показывает это в “панели 0”.

В твоём текущем окружении ты уже перевёл UI Proxy и React UI на systemd-сервисы (по твоим сообщениям). В публичном репо UI Proxy пока ориентирован на tmux-контроль — это нужно согласовать и привести к единой реальности через Art (см. “что ещё сделать”).

3.2 Каналы данных (кто что отправляет в Art)
Канал A: “ОС-уровень” (самый надёжный)

Источник: Art Agent
Что собирает:

journald/systemd логи и статусы юнитов REGART,

crashloop, exit code, причины падения,

port collisions,

сетевые пробы до LangGraph/UI Proxy/React UI.

Это закрывает 100% для случаев “проект не поднялся вообще”.

Канал B: “Бэкенд-уровень” (REGART сервисы как источники)

Источники: UI Proxy, LangGraph (и любые другие сервисы REGART)
Что нужно:

структурированные события/ошибки в формате RawEvent (Art) или через локальный мост (см. ниже).

корреляция (run_id/node_id/trace_id) если она реально известна, без угадайки.

Примечание про текущую реальность: сейчас UI Proxy в публичном репо проксирует /api/* и при connect error отдаёт простой JSON.
Для Art-интеграции UI Proxy должен стать источником “богатых” upstream-ошибок (не просто “недоступен”, а: что именно, куда, что делать, какие действия, какие доказательства), но так, чтобы при отсутствии Art UI Proxy продолжал работать.

Канал C: “Браузерный уровень” (самый близкий к пользователю)

Источник: Browser Level0 (JS)
Что уже есть в REGART:

перехват window.error и unhandledrejection,

fallback overlay,

единый буфер ошибок/событий, dedupe/throttle.

Что нужно добавить для Art:

“мост” (bridge): каждое важное событие Level0 → отправлять в Art Agent/Core (через ingest), с локальным backlog, чтобы при недоступном Art ничего не терялось.

4) “Глубокая интеграция” REGART ↔ Art (простыми словами)

“Глубокая интеграция” у нас означает не “Art встроен в LangGraph”, а вот что:

4.1 REGART отдаёт Art максимум правды о себе

REGART должен уметь:

сообщать Art о проблемах не только как строка, а как структурированное событие:

где случилось,

что именно случилось,

почему (если известно),

какое влияние,

что сделать (действия/инструкции),

доказательства (stack/log excerpt/HTTP meta).

4.2 Art даёт REGART единый “источник правды” по мониторингу

REGART UI (в перспективе) не должен “рисовать красное/жёлтое” сам по себе. Он должен:

либо отображать состояние из Art snapshot/stream,

либо отправлять raw_event и ждать, что Art решит — это incident или просто событие.

Это ровно совпадает с идеей Debugger-спеки REGART: “Level 0 = источник правды, Level 1 = экран”, только на уровне всей системы это расширяется до Art как “главного источника правды”.

5) Что уже реализовано в REGART (факт по публичному репо)
5.1 Debugger Level0 реально стартует до React и ловит фатальные ошибки

ui/src/main.jsx запускает Level0 до render.

Level0 хранит каналы errors/events/network/snapshots, dedupe/throttle, умеет copy bundle и показывает overlay.

Стоят ловушки error и unhandledrejection.

5.2 Level1 (React) уже пишет ошибки в Level0

core.js прямо говорит: Level0 = источник правды, Level1 только пушит/читает через Level0.

App.jsx пушит ошибки run-стрима в uiErrCore.

5.3 UI Proxy выполняет роль “единой точки входа” для UI

/api/* проксируется в LangGraph.

/ui/* даёт статус/управление LangGraph и UI Proxy, модели, probe tool_calls.

5.4 Graph UI уже умеет поднимать граф и стабилизировать отрисовку “пинком”

GraphView грузит /api/assistants/{id}/graph и делает fitView/resize-kick.
Но это пока не оформлено как события “UX-инцидентов” в Debugger/Art (например “граф пустой”, “контейнер 0×0”, “не отрисовался”) — это следующая стадия.

6) Что ещё нужно сделать, чтобы REGART стал полноценным источником для Art (без зависимости)

Ниже — именно “механизмы взаимодействия”, а не код.

6.1 Мост Level0 → Art Agent (главный недостающий механизм)

Добавить в Level0/Browser-часть:

Sender: фоновая отправка RawEvent в Art (локальный URL Art Agent).

Backlog: если Art недоступен — события копятся (IndexedDB) и догоняются позже.

observability_gap: если Art недоступен долго — фиксируем событие “Art unreachable”, чтобы было видно “дыра наблюдаемости”.

Цель: “ничего не теряем” при любых условиях.

6.2 Network provider в UI (чтобы реально было 100% по UI→backend)

Debugger-спека REGART требует ловить Network-канал (fetch wrapper) и держать request/response meta.
Сейчас в Level0 API для pushNetwork есть, но в публичном коде нет явного механизма автоматического перехвата всех fetch.
Нужно:

единый перехватчик сетевых запросов UI,

отправка network-событий в Level0 → Art.

6.3 UX-инциденты графа/верстки как события (чтобы “не отрисовалось” было фактом)

Debugger-спека REGART прямо говорит, что Graph-состояния и “пустой граф” должны попадать в Debugger.
Нужно:

датчики “контейнер 0×0”, “nodes=0”, “ReactFlow instance не готов”, “stale layout” и т.д.

всё это → raw_event → Art rules → incident.

6.4 “Богатые” upstream-ошибки UI Proxy (в формате, пригодном для Art)

В публичном UI Proxy сейчас ConnectError → JSON {error, base_url}.
Для Art нужно:

единый формат upstream-ошибок с полями “что/где/почему/что делать/действия/доказательства”;

и эти же ошибки должны:

показываться пользователю в REGART UI (как сейчас через Debugger),

уходить в Art как RawEvent.

Важно: REGART не должен ломаться, если Art не работает.

6.5 Корреляция (run_id/node_id/trace_id) — строго “только по данным”

Уже сейчас UI run-stream достаёт run_id из SSE события metadata.
Нужно:

чтобы все ошибки/события, связанные с конкретным запуском, добавляли run_id в контекст (если он реально известен),

чтобы события графа и tool/model событий не “угадывали” node_id (в Debugger-спеке прямо запрещён хардкод).
То есть: либо бэкенд отдаёт node_id/span_id, либо мы не делаем Jump “по эвристике”.

6.6 Сервисный слой (systemd/journald) — переезд управления из REGART внутрь Art

Сейчас UI Proxy содержит “управлялки” (старт/стоп/статус) для LangGraph и себя (tmux-ориентировано).
В целевой архитектуре:

“наблюдать и объяснять” должен Art,

“выполнять действия” (restart unit, показать логи) — через Art Actions + агент (с аудитом).

REGART может оставить кнопки “перезапустить”, но они должны вызывать Art action, а не жить отдельной логикой.

7) Как это выглядит для пользователя (сценарии)
Сценарий 1: всё работает

Пользователь открывает REGART UI.

Level0 уже активен до React и ловит любые фатальные ошибки.

Все события (ошибки/сеть/UX) уходят в Art → Art строит инциденты → в Panel0 видно “зелёное/жёлтое/красное”.

Сценарий 2: LangGraph упал/не запущен

UI Proxy при проксировании /api/* ловит ConnectError → отдаёт ошибку.

UI фиксирует это через Debugger/Art как upstream_unavailable.

Art Agent одновременно видит по systemd/journald реальную причину (unit down, crashloop, порт занят).

Сценарий 3: React UI вообще не стартует (Vite/сборка/порт)

REGART UI недоступен.

Но Art всё равно показывает причину в Panel0 (лог systemd unit, порт занят, build error), потому что Art Agent это собрал на уровне ОС.

Сценарий 4: Art недоступен

REGART продолжает работать как сейчас: Level0 держит локальный буфер и overlay.

Параллельно события копятся в backlog и будут доставлены в Art позже (после восстановления).
