# Backup/restore SQLite

## Назначение
Этот документ описывает единственный допустимый способ резервного копирования (backup, резервная копия) и восстановления
SQLite-состояния `Art Core` в эксплуатационном контуре `stage23`.

Речь идёт не о произвольной SQLite рядом с системой, а о **реальной рабочей базе Core**:
- события `v1/v2`;
- инциденты;
- аудит;
- аналитика;
- производные состояния, уже переведённые в durable-basement.

Ключевое правило:
- резервируется именно тот файл, который `art-core` использует как `CORE_DB_PATH`;
- проверка после восстановления тоже идёт на этом же рабочем `CORE_DB_PATH`.

## Почему используется именно `.backup`
- Команда `sqlite3 ".backup"` корректно работает с SQLite и учитывает WAL.
- Это безопаснее, чем копировать файл БД “как есть” в произвольный момент.
- Для `Art` это канонический путь резервного копирования на уровне runbook и CI-smoke.

## Обязательный порядок
1. Остановить `Art Core`.
2. Выполнить backup рабочей базы:

```bash
sqlite3 /path/to/core.sqlite3 ".backup '/path/to/backups/core-YYYYMMDD-HHMMSS.sqlite3'"
```

3. Выполнить проверку целостности backup-файла:

```bash
sqlite3 /path/to/backups/core-YYYYMMDD-HHMMSS.sqlite3 "PRAGMA integrity_check;"
```

Ожидаемый результат:

```text
ok
```

4. При восстановлении:
   - удалить или изолировать повреждённый runtime-файл БД;
   - вернуть backup на исходный рабочий путь;
   - снова выполнить `PRAGMA integrity_check`;
   - только потом запускать `Art Core`.

## Что именно должно пережить restore
После восстановления обязаны сохраниться:
- события, уже принятые системой;
- инциденты и их состояния;
- audit chain (цепочка аудита);
- аналитические сводки, если они уже были материализованы в SQLite;
- производные runtime-состояния, для которых закрыт `stage11`.

## Что считается PASS
Backup/restore считается корректным только если одновременно выполняются все условия:
- backup-файл реально создан;
- `integrity_check` на backup-файле возвращает `ok`;
- восстановленный файл снова проходит `integrity_check`;
- после запуска `Core` snapshot, incidents, audit verify и analytics подтверждают, что рабочее состояние пережило restore.

## Runtime proof
Боевой smoke для этого контура:

```bash
bash scripts/tests/ops_stage23_smoke.sh
```

Этот smoke теперь делает именно следующее:
- поднимает живой `art-core` на временной рабочей БД;
- создаёт реальное состояние Core через ingest + audit;
- останавливает Core;
- делает `.backup` настоящей рабочей БД Core;
- имитирует потерю узла;
- восстанавливает БД на исходный путь;
- снова поднимает Core;
- проверяет snapshot, incidents, audit verify и analytics.

## Pass/fail
- `pass`: все проверки выше выполнены.
- `fail`: любое отклонение, включая “backup создался, но восстановленное состояние не совпадает”.
