## Статус
Этот документ является универсальным шаблоном модели:
- `ideation` (формирование замысла);
- `governance` (управление проектом и правила принятия решений);
- `remediation` (исправление дефектов и доведение системы до нужного состояния).

Его можно накладывать на любой инженерный проект как канон построения, контроля качества, аудита, исправления дефектов и принятия решений.

Он не заменяет конкретную спецификацию проекта. Он задаёт общий каркас, на который потом насаживаются предметная модель, архитектура, чек-листы, тесты, релизный процесс и эксплуатация.

## 1. Назначение
Этот шаблон нужен, чтобы проект:
- строился не хаотично, а от корня к производным слоям;
- не жил на локальных догадках и случайных решениях;
- не закрывал этапы формально;
- не разваливался при росте объёма кода, людей, окружений и интеграций;
- был пригоден к продакшену во враждебной среде;
- был доказуемым, управляемым и воспроизводимым.

## 2. Главная модель: дерево проекта
Любой серьёзный проект строится как дерево решений.

### 2.1. Корень
Корень задаёт смысл, неизменяемые законы и целевую природу системы.

В любом проекте в корень должны входить минимум четыре документа:
1. `Constitution / Foundation` (`Конституция / Основание`)
2. `History / Concepts` (`История / Концепции`)
3. `System Spec / Product Spec` (`Системная спецификация / Продуктовая спецификация`)
4. `External Integration Spec` (`Спецификация внешних интеграций`)

Назначение этих документов:

### 2.1.1. Constitution / Foundation (`Конституция / Основание`)
Это главный закон проекта.

Он отвечает на вопросы:
- какие принципы запрещено нарушать;
- что в проекте считается истиной;
- какие архитектурные и процессные инварианты обязательны;
- какие anti-patterns (`запрещённые стили работы`) недопустимы.

Если говорить простыми словами, это “основной закон государства” внутри проекта.

### 2.1.2. History / Concepts (`История / Концепции`)
Это документ, который объясняет:
- как родился проект;
- зачем он нужен;
- какие идеи и находки уже были утверждены;
- что делает проект особенным;
- какие сильные стороны нельзя потерять при развитии.

Простыми словами: это “смысловая память проекта”.

### 2.1.3. System Spec / Product Spec (`Системная / Продуктовая спецификация`)
Это документ, который переводит замысел в предметную модель и требования.

Он должен объяснять:
- из каких сущностей состоит система;
- как они взаимодействуют;
- какие контракты, данные, состояния и ограничения допустимы;
- как должна вести себя система для пользователя и для оператора.

Простыми словами: это “чертёж того, что именно мы строим”.

### 2.1.4. External Integration Spec (`Спецификация внешних интеграций`)
Это документ о том, как проект живёт рядом с другими системами.

Он должен объяснять:
- откуда приходят данные;
- куда они уходят;
- как устроен обмен сигналами;
- где границы ответственности;
- как не потерять безопасность, корректность и управляемость на стыках.

Простыми словами: это “правила общения проекта с внешним миром”.

Корень отвечает на вопросы:
- что это за система;
- чем она обязана быть;
- какие законы нельзя нарушать;
- какие differentiators нельзя потерять;
- с какими внешними системами и по каким правилам она взаимодействует.

### 2.2. Ствол
Ствол превращает смысл проекта в порядок исполнения.

Ствол должен состоять из последовательности:
1. полный аудит;
2. дефектовочная контрольная ведомость;
3. дефектовочная лестница remediation;
4. `MASTER` checklist.

Ствол делает три вещи:
- показывает расхождение между корнем и фактическим состоянием;
- задаёт corrective-порядок;
- превращает этот порядок в исполнимую программу.

`MASTER` всегда завершает ствол. Выше `MASTER` не может быть документа, который управляет производными слоями напрямую, минуя ствол.

После полного аудита ствол обязан содержать не только лестницу исправления, но и прямую defect-to-stage контрольную ведомость. Без неё remediation остаётся слишком общим и не гарантирует, что каждый найденный недостаток будет реально отработан.

Важно:
- закрытие нижнего corrective-слоя не означает автоматического закрытия всего дефекта;
- если нижний слой честно доведён, активный corrective-контур обязан смещаться на следующий downstream stage и быть явно отражён в обзорных документах;
- проект не имеет права продолжать рассказывать, что активный baseline остаётся в уже закрытом нижнем слое.

### 2.3. Крона
Крона проекта:
- `stage-checklists` (`этапные чек-листы`) — пошаговые исполнительные программы по этапам проекта;
- код — сама реализованная система;
- тесты — автоматические проверки, доказывающие корректность поведения;
- `CI/CD` (`непрерывная интеграция / непрерывная доставка`) — автоматическая проверка, сборка и подготовка поставки;
- `runtime` (`рабочее исполнение`) — то, как система реально ведёт себя в запущенном состоянии;
- `contracts` (`контракты`) — схемы, форматы, API и правила обмена данными;
- `docs` (`документация`) — объяснение того, как система устроена, используется и сопровождается;
- `runbooks` (`операционные инструкции`) — пошаговые инструкции для инцидентов, деградаций и штатных операций;
- `release artifacts` (`артефакты релиза`) — пакеты, архивы, образы, подписи, контрольные суммы и всё, что реально поставляется;
- `evidence` (`доказательные артефакты`) — логи, отчёты, скриншоты, выводы команд и другие материалы, доказывающие реальное состояние проекта.

Простыми словами, крона — это весь “видимый и исполняемый проект”: всё, чем пользуются разработчики, операторы, покупатели, тесты и production-среда.

Крона не имеет права противоречить стволу и корню.

## 3. Закон №1: движение от корня по стволу
Любое действие должно делаться по пути:

`корень -> ствол -> нужная точка кроны`

Это означает:
1. Нельзя начинать работу с локального файла, не подняв в контекст корень.
2. Нельзя менять крону, если не проверено, не ломает ли это ствол.
3. Нельзя принимать локальное решение только потому, что оно “удобно здесь и сейчас”.
4. Если локальная задача конфликтует с корнем, побеждает корень.
5. Если локальная задача конфликтует со стволом, побеждает ствол.

## 4. Базовые законы универсального проекта

### 4.1. Закон буквального исполнения
Любое утверждённое требование должно пониматься как обязательное действие.

Запрещено:
- трактовать приказ как пожелание;
- снижать объём работы “потому что и так понятно”;
- самовольно сокращать глубину проверки;
- заменять требуемое действие более лёгким аналогом.

### 4.2. Закон максимализма
Проект делается не “достаточно хорошо”, а по верхней границе качества.

Это означает:
- максимальную релевантную глубину анализа;
- максимальную релевантную глубину дебаггинга;
- максимальное покрытие угроз;
- максимальную честность в оценке незавершённости.

Запрещены формулировки и подходы:
- “для pass достаточно”;
- “минимальный smoke”;
- “пока и этого хватит”;
- “почти готово”.

### 4.3. Закон доказанного эксплуатационного эффекта
Любое изменение считается завершённым только после доказанного эксплуатационного эффекта.

Эффект должен быть подтверждён:
- не одним удачным симптомом;
- не одной командой;
- не одной картинкой;
- а разносторонним дебаггингом.

Нужно доказать:
- что именно исправилось;
- почему это исправилось;
- какие альтернативные причины исключены;
- что исправление не породило скрытый дефект в другом слое.

### 4.4. Закон hostile production
Production считается враждебной средой.

Система должна проектироваться так, как будто её пытаются сломать:
- сетью;
- перегрузкой;
- отказами зависимостей;
- неверной конфигурацией;
- человеческой ошибкой;
- злоупотреблением правами;
- плохими входными данными;
- несовместимостью окружения;
- частичным отказом;
- утечкой секрета;
- supply-chain дефектом;
- регуляторным конфликтом;
- деградацией UX в критический момент.

### 4.5. Закон спуска к корневой причине
Если дефект найден на каком-либо уровне, перед исправлением симптома необходимо спуститься ниже.

Алгоритм:
1. Найти место проявления дефекта.
2. Опуститься на уровень ниже.
3. Полностью проверить основание.
4. Если найден более глубокий источник — опуститься ещё ниже.
5. Остановиться только тогда, когда устранена корневая причина.

### 4.6. Закон authenticity
Проект должен быть аутентичным и безопасным с точки зрения авторских прав.

Запрещено:
- встраивать спорные чужие материалы без доказанного права;
- использовать визуальные, текстовые, звуковые, кодовые или контентные сущности, которые могут привести к спору;
- выдавать за собственные артефакты, происхождение которых не доказано.

Обязательны:
- allowlist допустимых внешних сущностей;
- provenance происхождения;
- политика authenticity;
- контроль drift и случайных включений.

### 4.7. Закон документационной синхронизации
Документы не имеют права устаревать независимо друг от друга.

Если меняется документ корня, должны синхронно меняться зависимые документы ствола и кроны.

Обязательны:
- machine-readable dependency map;
- CI-gate на рассинхронизацию;
- запрет слияния drift-состояния.

### 4.7A. Закон человекочитаемой документации
Документация в любом зрелом проекте должна писаться так, чтобы её мог понять не только автор и не только узкий специалист.

Это означает:
- почти все документы должны читаться человеком без полного знания внутренней истории проекта;
- сложные термины должны расшифровываться;
- англоязычные слова и сокращения должны переводиться или поясняться;
- документ должен объяснять смысл, а не только фиксировать факт;
- документ должен быть пригоден для сопровождения через месяцы и годы.

Простыми словами:
- документация должна помогать разобраться;
- не должна заставлять догадываться;
- не должна требовать "быть внутри головы автора".

Правильный документ отвечает минимум на вопросы:
1. Что это?
2. Зачем это нужно?
3. Как это работает?
4. Что сломается или изменится, если это нарушить?

Документ считается слабым, если:
- он написан внутренним жаргоном без расшифровки;
- он понятен только тому, кто уже знает код;
- он перечисляет действия, но не объясняет причин;
- он использует английские специальные слова без перевода или пояснения;
- он не помогает новому инженеру, оператору, аудитору или заказчику понять систему.

Подробность здесь считается достоинством, а не недостатком, если она:
- снижает риск неправильного понимания;
- упрощает сопровождение;
- ускоряет ввод нового участника;
- уменьшает вероятность ложных изменений и ошибок эксплуатации.

### 4.8. Закон честного статуса
Никакой этап, шаг, чекбокс или артефакт не считается закрытым, пока:
- не выполнена работа;
- не получена проверка;
- не получен артефакт результата;
- не исключён формальный false-green.

## 5. Что должно входить в корень любого проекта

### 5.1. Foundation / Constitution
`Foundation / Constitution` (`Основание / Конституция`) — это главный документ проекта, в котором нельзя писать пожелания или наброски. Там фиксируются только законы, инварианты и обязательные правила.

Это документ, который отвечает не на вопрос “как удобно сделать”, а на вопрос “что запрещено ломать”.

Должен содержать:
- природу системы;
- основные законы;
- `Truth/Reality model` (`модель истины и реальности системы`);
- архитектурные инварианты;
- `UX/ops/security/perf/privacy` принципы (`пользовательский опыт / эксплуатация / безопасность / производительность / приватность`);
- `anti-patterns` (`запрещённые стили работы и плохие практики`);
- правила релиза и проверки.
- принцип человекочитаемой документации и запрет на документацию "только для автора".

### 5.2. History / Concepts
`History / Concepts` (`История / Концепции`) нужен, чтобы проект не потерял собственный смысл при долгой разработке.

Именно здесь фиксируется:
- что было придумано;
- что было утверждено;
- какие идеи обязательны;
- какие идеи экспериментальны;
- какие идеи нельзя случайно “оптимизировать” до исчезновения.

Должен содержать:
- историю замысла;
- все утверждённые `differentiators` (`уникальные отличия проекта`);
- объяснение, почему проект вообще существует;
- что делает систему уникальной;
- какие идеи обязательны, а какие экспериментальны.

### 5.3. System Spec / Product Spec
`System Spec / Product Spec` (`Системная / Продуктовая спецификация`) — это рабочий чертёж проекта.

Этот документ нужен, чтобы любой инженер, тестировщик, аналитик или оператор понимал:
- что система должна делать;
- из чего она состоит;
- где проходят границы модулей;
- какие состояния и сценарии допустимы;
- какие требования обязательны.

Должен содержать:
- предметную модель;
- сущности;
- контракты;
- поведенческую модель;
- ограничения;
- требования к данным, `transport` (`передаче данных`), `runtime` (`рабочему исполнению`), `UX` (`пользовательскому опыту`), `audit` (`аудиту и доказуемости`).

### 5.4. External Integration Spec
`External Integration Spec` (`Спецификация внешних интеграций`) описывает стыки проекта с другими системами.

Без этого документа проект быстро начинает жить на догадках:
- кто кому что отправляет;
- где истина;
- где граница ответственности;
- что считать ошибкой интеграции;
- как переживать сетевые, форматные и организационные разрывы.

Должен содержать:
- внешние системы;
- модели интеграции;
- границы ответственности;
- `data flow` (`поток данных`);
- `policy/security` ограничения (`политики и ограничения безопасности`);
- корреляцию сигналов;
- `transport` (`каналы передачи`) и `failure modes` (`режимы отказа и деградации`).

## 6. Что должно входить в ствол

### 6.1. Полный аудит
Должен отвечать:
- какие файлы существуют;
- что реально покрыто;
- где `OK`, `WEAK`, `MISMATCH`;
- что противоречит корню;
- где симптом, а где источник.

### 6.2. Дефектовочная ведомость
Должна отвечать:
- что чинить;
- в каком порядке;
- почему нельзя прыгать выше;
- какие нижние слои блокируют верхние.

### 6.3. MASTER checklist
Должен отвечать:
- как превратить corrective-порядок в исполнимую программу;
- какой этап выполняется первым;
- какой этап нельзя начинать, пока не закрыт предыдущий;
- когда проект реально готов к продакшену.

## 7. Что должно входить в крону
Крона должна быть производной от корня и ствола.

В крону входят:
- `stage-checklists` (`этапные чек-листы`) — последовательные программы исполнения по крупным частям проекта;
- код;
- тесты;
- `runbooks` (`операционные инструкции`) — готовые пошаговые сценарии, что делать при штатных и аварийных ситуациях;
- `release pipeline` (`релизный конвейер`) — процесс сборки, проверки, подписи и выпуска версии;
- `CI/CD` (`непрерывная интеграция / непрерывная доставка`) — автоматический контур проверок и подготовки поставки;
- платформенные профили;
- `UI/UX` слой (`пользовательский интерфейс и пользовательский опыт`);
- `analytics` (`аналитика`) — производные сводки, агрегаты и метрики, помогающие понимать поведение системы;
- `observability` (`наблюдаемость`) — сигналы, логи, события, метрики и трассы, которые позволяют видеть реальное состояние системы;
- `evidence corpus` (`корпус доказательных артефактов`) — весь массив материалов, доказывающих поведение системы, результаты проверок и факты расследования.

Простыми словами, крона — это всё то, что люди реально видят, запускают, проверяют, поддерживают и поставляют.

Для каждого элемента кроны должно быть ясно:
- на какой root-document он опирается;
- на какой шаг `MASTER` он опирается;
- какие угрозы он обязан выдерживать;
- какой evidence доказывает его работоспособность.

## 8. Универсальный алгоритм работы по проекту

### 8.1. Перед началом любого действия
1. Поднять в контекст корень.
2. Поднять в контекст ствол.
3. Определить нужную точку кроны.
4. Проверить, нет ли конфликта.
5. Только потом действовать.

### 8.1A. Перед изменением документа
1. Проверить, относится ли документ к корню, стволу или кроне.
2. Проверить, какие зависимые документы должны измениться вместе с ним.
3. Обновить не только технический факт, но и объяснение смысла.
4. Убедиться, что новый текст остаётся понятным человеку без узкоспециализированного контекста.
5. Проверить, что сложные термины и сокращения объяснены.

### 8.2. Перед исправлением дефекта
1. Зафиксировать симптом.
2. Спуститься на уровень ниже.
3. Найти корень.
4. Исправить корень.
5. Перепроверить вверх по слоям.

### 8.3. Перед закрытием шага
1. Проверить факт выполнения.
2. Проверить hostile path.
3. Проверить negative path.
4. Проверить отсутствие альтернативной причины.
5. Проверить артефакт результата.
6. Проверить, что не появился новый дефект в соседнем слое.

## 9. Универсальная модель тестирования
Тестовый корпус должен иметь несколько обязательных уровней:

1. `Contract` (`контрактный уровень`)
2. `Behavior` (`поведенческий уровень`)
3. `Integration` (`интеграционный уровень`)
4. `Operational` (`эксплуатационный уровень`)
5. `Adversarial` (`враждебный / атакующий уровень`)
6. `Regression` (`анти-регрессионный уровень`)

Расшифровка:

### 9.1. Contract (`Контрактный уровень`)
Проверяет, что форматы, схемы, API и обязательные поля соответствуют договорённости.

Простыми словами: “данные и интерфейсы не разъехались”.

### 9.2. Behavior (`Поведенческий уровень`)
Проверяет, что функция делает именно то, что должна делать по смыслу.

Простыми словами: “не только формат правильный, но и логика правильная”.

### 9.3. Integration (`Интеграционный уровень`)
Проверяет, как работают вместе несколько частей системы.

Простыми словами: “модуль по отдельности хороший, но умеет ли он жить с соседями”.

### 9.4. Operational (`Эксплуатационный уровень`)
Проверяет, как система ведёт себя в условиях, близких к реальной эксплуатации:
- запуск;
- конфиги;
- доставка;
- восстановление;
- деградация;
- обслуживаемость.

Простыми словами: “можно ли этим реально пользоваться и сопровождать”.

### 9.5. Adversarial (`Враждебный / атакующий уровень`)
Проверяет не удобный happy-path, а неприятные условия:
- плохой ввод;
- обрывы связи;
- частичные отказы;
- попытки обойти ограничения;
- конфликтные состояния;
- hostile production scenarios.

Простыми словами: “что будет, если среда или пользователь начнут ломать систему”.

### 9.6. Regression (`Анти-регрессионный уровень`)
Проверяет, что старое рабочее поведение не сломалось после новых изменений.

Простыми словами: “новое исправление не убило старую правильную функцию”.

Для важных функций недостаточно только `Contract`.

Если нет как минимум:
- `integration` (`интеграционного теста, который проверяет работу с соседними частями системы`);
- `negative path` (`негативного сценария`, где что-то идёт не по плану: неверный ввод, ошибка, отказ, пустые данные);
- `hostile/adversarial path` (`враждебного сценария`, где система сталкивается с агрессивной, конфликтной или разрушительной средой);

то функция считается недодоказанной.

## 10. Универсальная модель артефактов
Каждый значимый этап должен оставлять:
- файл;
- лог;
- отчёт;
- вывод команды;
- screenshot или runtime artifact;
- запись в evidence/ledger.

Без артефакта нет доказательства.
Без доказательства нет закрытия.

## 11. Универсальная модель release-governance
В проекте должны быть:
- `release process` (`процесс релиза`) — кто, когда, как и при каких условиях выпускает новую версию;
- `go/no-go template` (`шаблон решения “выпускать / не выпускать”`) — формальная точка принятия решения перед релизом;
- `rollback plan` (`план отката`) — что делать, если новый выпуск нужно быстро и безопасно отменить;
- `compatibility matrix` (`матрица совместимости`) — с какими платформами, версиями и окружениями система совместима;
- `delivery evidence` (`доказательства поставки`) — артефакты, подтверждающие, что выпуск реально собран, проверен и готов;
- `production debt register` (`реестр производственного долга`) — список оставшихся рисков и слабых мест, которые сознательно не закрыты;
- `security posture` (`картина безопасности`) — в каком состоянии находятся угрозы, защиты, ограничения и доверительные границы;
- `authenticity policy` (`политика аутентичности`) — правила, не позволяющие тащить в проект спорные чужие сущности и недоказанное происхождение.

Финальный `MASTER` может закрываться только в момент реального `Production GO`.

## 12. Универсальная модель anti-patterns
Запрещено как стиль работы:
- `hardcoding` (`жёсткое вшивание`) — когда адреса, правила, строки, политики или значения зашиты прямо в код/документы без конфигурируемого и проверяемого основания;
- `placeholder-driven development` (`разработка на заглушках`) — когда проект выглядит готовым, но в основе лежат временные фиктивные сущности;
- `false-green testing` (`ложно-зелёное тестирование`) — когда тесты зелёные, но не доказывают реальную работоспособность;
- `cosmetic closure` (`косметическое закрытие`) — когда этап закрыт по бумаге, а не по факту;
- `docs stronger than runtime` (`документация сильнее рантайма`) — когда в документах обещано больше, чем реально умеет система;
- `runtime stronger than policy` (`рантайм сильнее политики`) — когда код делает больше, чем разрешено и зафиксировано правилами;
- `drift между корнем и производными` (`рассинхронизация`) — когда нижние документы и код уже живут отдельно от корня;
- `дублирующие источники истины` — когда один и тот же закон описан в нескольких местах по-разному;
- скрытые `assumptions` (`предположения`) без явной фиксации — когда проект держится на неописанных “само собой разумеется”;
- магические `implicit rules` (`неявные правила`) — когда важная логика существует, но нигде явно не закреплена;
- пропуск `root-cause` анализа (`поиска корневой причины`) — исправление симптома вместо источника проблемы;
- выборочный аудит вместо полного — когда проверяется только удобный кусок проекта, а не весь нужный контур;
- “сначала сделаем, потом додумаем” — когда система строится без корня и ствола;
- “это только временно” в baseline проекта — когда временное решение превращается в постоянную слабость;
- зависимость на одного человека без фиксации `bus-factor` риска (`риска потери критического знания вместе с одним человеком`).

## 13. Минимальный обязательный файловый каркас для любого проекта
В универсальном проекте должен существовать минимум такой корпус:

### Корень
- `FOUNDATION_CONSTITUTION.md`
- `PROJECT_HISTORY_AND_CONCEPTS.md`
- `SYSTEM_SPEC.md`
- `EXTERNAL_INTEGRATION_SPEC.md`

### Ствол
- `FULL_AUDIT_PROGRAM.md`
- `FULL_AUDIT_REGISTRY.md`
- `DEFECT_REMEDIATION_LADDER.md`
- `MASTER_CHECKLIST.md`

### Крона
- `CHECKLIST_XX_*.md`
- `contracts/*`
- `tests/*`
- `scripts/ci/*`
- `docs/ops/*`
- `docs/security/*`
- `docs/privacy/*`
- `docs/release/*`
- `evidence/*`

## 14. Как накладывать этот шаблон на новый проект
1. Создать 4 root-document.
2. Создать machine-readable dependency map корня.
3. Создать CI-gate на sync root-dependents.
4. Провести полный line-by-line аудит.
5. Создать дефектовочную ведомость.
6. Построить `MASTER`.
7. Только после этого строить stage-checklists и код.
8. Любое расширение проекта делать только через дерево:
   - корень;
   - ствол;
   - крона.

## 15. Definition of Done для этой модели
Модель применена правильно только если:
- корень определён явно;
- ствол определён явно;
- крона подчинена стволу;
- документы не дрейфуют;
- аудит завершён;
- дефектовочная лестница построена;
- `MASTER` отражает ствол;
- ни один этап не закрывается формально;
- продакшен-готовность доказывается evidence, а не словами.
