# Политика Error Budget

## Назначение

Error budget нужен не для отчётности, а для ограничения риска.  
Если бюджет ошибок исчерпан, проект обязан временно снизить долю рискованных изменений до восстановления устойчивости.

## Правило заморозки

При исчерпании error budget:

- замораживаются рискованные изменения в `main`;
- release кандидаты, повышающие риск, блокируются;
- разрешаются только изменения, необходимые для восстановления стабильности.

## Что считается разрешённым во время freeze

- bugfix для активного инцидента;
- security fix;
- rollback;
- observability / evidence / diagnostics improvements;
- документированный fix для SLO breach или `observability_gap.*`.

## Что запрещено во время freeze

- новые feature changes;
- рискованные refactor без прямой связи с инцидентом;
- изменение contracts без emergency justification;
- визуальные/маркетинговые изменения, не влияющие на восстановление сервиса.

## Снятие заморозки

Заморозка снимается только если одновременно:

- две последовательные SLO windows зелёные;
- нет активного `SEV0/SEV1`;
- backlog восстановлен до нормы;
- on-call подтверждает снятие freeze;
- release owner фиксирует решение в change/release trail.

## Критерий актуальности

Документ считается актуальным только если:

- зафиксирована freeze policy;
- перечислены разрешённые типы изменений;
- перечислены запрещённые типы изменений;
- определены критерии снятия freeze.
